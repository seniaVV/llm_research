## Инструкция по запуску

Для воспроизведения результатов данного исследования необходимо выполнить следующие шаги в терминале:

1. Подготовка окружения и зависимостей

```
# Создание и активация виртуального окружения
python3 -m venv .venv
source .venv/bin/activate

# Установка необходимых библиотек
pip install -r requirements.txt
```

2. Запуск инфраструктуры (Docker)

```
# Запуск PostgreSQL, MinIO и Open WebUI
docker-compose up -d

# Проверка статуса контейнеров
docker-compose ps
```

3. Импорт тестовых данных

```
# Импорт схемы, метаданных и S3-объектов в локальные сервисы
ENV_FILE=.env ./agrometrics_ai_full_handoff_3_fields_20260701/scripts/import_local.sh .

# Проверка корректности импорта
ENV_FILE=.env ./agrometrics_ai_full_handoff_3_fields_20260701/scripts/verify_local.sh .
```
4. Запуск скриптов исследования модели

```
# Тест 1: Подбор параметров генерации (Grid Search)
python -m src.01_test_parameters

# Тест 2: Подключение к БД методом Ручного RAG (Psycopg2 + OpenAI API)
python -m src.02_method_openai

# Тест 3: Подключение к БД методом Text-to-SQL (LangChain)
python -m src.03_method_langchain

# Тест 4: Проверка на галлюцинации и краевые сценарии
python -m src.04_test_hallucinations
```

## Развертывание сервисов в Docker

Файл docker-compose.yml состоит из 4-х ключевых сервисов:
1. `PostgreSQL + PostGIS (postgis/postgis:15-3.3)`: Основная база данных для хранения структурированных метаданных полей, их геометрии, погодных условий и связей с растровыми активами.
2. `MinIO (minio/minio:latest)`: S3-совместимое объектное хранилище для тяжелых файлов (GeoTIFF, спутниковые снимки Sentinel-1/2, индексы).
3. `MinIO-init (minio/mc:latest)`: Служебный контейнер, который при запуске автоматически создает бакет agrometrics-raw и настраивает права доступа к нему.
4. `Open WebUI (open-webui:main)`: Графический веб-интерфейс для взаимодействия пользователя с локальной языковой моделью (Qwen2.5:14b), подключенный к запущенному на хост-машине серверу Ollama.

## Исследование модели и подключение к БД

**Подбор параметров генерации**
```
python -m src.01_test_parameters
```

*Методология:* Автоматизированный поиск оптимальной конфигурации гиперпараметров модели Qwen2.5:14b с использованием алгоритма `Grid Search` и кастомной эвристической функции оценки (scoring function).

*Описание параметров генерации*

| Параметр | Назначение | Влияние на вывод |
|----------|------------|------------------|
| `temperature` | Регулирует степень случайности при выборе следующего слова. | Низкие значения (0.1–0.3) обеспечивают детерминированный, фактологический вывод. Высокие (0.7–1.0) повышают креативность, но увеличивают риск галлюцинаций. |
| `top_p` | Динамическое ограничение словаря. Модель рассматривает только токены, чья суммарная вероятность достигает порога `p`. | Отсекает низковероятные "хвосты" распределения. Значения 0.8–0.9 обеспечивают естественность языка при сохранении релевантности. |
| `top_k` | Жесткое ограничение количества кандидатов. Модель выбирает только из `k` наиболее вероятных токенов. | Работает в паре с `top_p`. Значения 20–40 предотвращают выбор семантически нерелевантных токенов. |
| `num_predict` (max_tokens) | Абсолютный лимит длины генерации в токенах. | Фиксирует верхнюю границу вывода. Оптимален для баланса между полнотой ответа и скоростью инференса. |

*Конфигурация поиска*

- Пространство поиска: 36 комбинаций гиперпараметров (3 × 3 × 2 × 2).
- Тестовый промпт: 
```
"Кратко объясни, что означает NDVI = 0.45 для пшеницы в фазе колошения. "
"Перечисли 2-3 конкретных агрономических риска и одну рекомендацию. "
"Не выдумывай точные цифры урожайности или норм внесения, если они не даны."
```

*Эвристическая функция оценки*

Максимальный балл: 100  

Базовая оценка: 50

| Критерий | Влияние на скор | Обоснование |
| :--- | :--- | :--- |
| Наличие доменных терминов (пшеница, стресс, азот, влага и др.) | +4 за каждый (макс. +20) | Проверка релевантности ответа предметной области |
| Выдумывание конкретных числовых значений (урожайность, нормы) | −25 | Детекция галлюцинаций — критический штраф |
| Повторы фраз | −5 за каждый повтор | Штраф за деградацию качества генерации |
| Длина ответа > 1100 символов | −5 и более | Штраф за избыточную генерацию |
| Длина ответа < 200 символов | −30 | Штраф за отказ от генерации или тривиальный ответ |
| Оптимальная длина (400–900 символов) | +7 | Поощрение лаконичности и полноты |
| Наличие структурных маркеров (списки, нумерация) | +5 | Поощрение читаемости и структурированности |
| Упоминание рисков (2-3 раза) | +3 до +10 | Поощрение выявления агрономических проблем |
| Наличие рекомендации | +3 | Практическая ценность ответа |
| Высокая плотность информации | +1 до +5 | Оценка соотношения терминов к длине текста |

## Заключение

**Достоинства:**
* Высокая фактологическая точность в парадигме RAG: При получении структурированных данных (JSON) модель отлично интерпретирует агрономические индексы (NDVI, NDMI), связывает их с погодными условиями и формулирует взвешенные рекомендации без галлюцинаций.
* Честность при отсутствии данных: Модель не пытается "угодить" пользователю выдуманными цифрами, если в контексте есть null или отсутствующие записи.
* Способность к логическому анализу: Успешно выявляет и объясняет противоречия в данных, предлагая гипотезы, а не просто констатируя факты.
* Эффективность: 14 миллиардов параметров обеспечивают оптимальный баланс между качеством рассуждений и скоростью локального инференса.

**Ограничения:**
* Ненадежность Text-to-SQL: Прямая генерация SQL-запросов моделью 14B для схемы с JSONB-полями приводит к синтаксическим ошибкам, выдумыванию колонок и нарушению формата вывода.
* Зависимость от качества промпта: Для стабильной работы требуется жесткая системная инструкция, явно запрещающая выдумывание данных, иначе модель может добавить общие, нерелевантные фразы.
* Отсутствие мультимодальности: Модель анализирует только текстовые метаданные и статистики. Для анализа самих GeoTIFF-изображений требуется предварительная экстракция признаков.
